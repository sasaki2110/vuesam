# Phase 3: 画面の違いを Spec だけで表現できるようにする

## このフェーズの目的

**一言で言うと**: ヘッダ項目、Enter の移動順、F キーの割り当てまで、すべて設定データ（Spec）で定義できるようにして、**新しい業務画面 = 新しい Spec ファイル**の状態にする。

**背景**: Phase 2 までで、グリッドの列定義と値確定の副作用は Spec 化できています。しかし、まだ以下が Vue テンプレートにハードコードされています:
- ヘッダの入力項目（契約先・納入先・納期など）の種類や並び順
- ヘッダからグリッドへの Enter 移動の流れ
- F キーに何の機能を割り当てるか

Phase 3 では、これらもすべて Spec に移します。結果として、新しい業務画面（発注・生産計画など）を作るとき、原則として **Vue の薄いラッパー + 新しい Spec ファイル** だけで済む状態になります。

---

## 前提

- Phase 2 が完了していること（field type レジストリと commitSpec の汎用化が終わっている）

---

## 完了条件

1. **ヘッダ項目が Spec で定義されている**
   - ヘッダにフィールドを追加したり順番を変えたりするとき、Vue テンプレートを編集する必要がない
   - テンプレートは `v-for` で Spec から動的に描画する形（1か所のみ）

2. **Enter の移動順が Spec で定義されている**
   - グリッド内の Enter 巡回列が `navigationSpec`（または gridSpec から導出されるデータ）として定義されている
   - Phase 1 で導入した `editChainColIds` 配列がここに統合されている

3. **F キーの割り当てが Spec で定義されている**
   - `keySpec` から「F1 → 新規、F12 → 保存」のようなマッピングが取得できる
   - 画面固有の F キー（F07 など）を Spec で追加・上書きできる
   - 未割り当てのキーは何もしない（方針を `decisions.md` に記載）

4. **ルーティングが業務中立のキーを使っている**
   - 例: `meta.screenSpecId: 'order-new'` のように、画面の種類を問わない形に寄せる
   - 完全なリネームは段階的でよいが、Phase 3 終了時に方針を決める

---

## Phase 3 を始める前に決めておくこと

`screen-spec-plan.md` セクション 8 の未確定事項のうち、以下の2つを **決めて `decisions.md` に記録してから**実装に入る。

| 決定事項 | 選択肢 |
|----------|--------|
| **Enter 順の表現方法** | A) 配列で順番に並べる / B) フィールド ID → 次の ID のマップ |
| **画面固有ハンドラの接続方法** | A) Spec から import する / B) 画面マウント時に関数注入する（テストしやすい） |

## 作業手順

### ステップ 3.1: ヘッダ項目を Spec 化する

**`headerSpec` の型を定義する**（最小限から始める）:
```typescript
type HeaderField = {
  id: string           // フィールドID（例: 'contractParty'）
  label: string        // 表示ラベル（例: '契約先'）
  editorType: string   // 入力部品の種類（例: 'masterCombobox'）
  editorProps?: object // 入力部品に渡す追加設定
  gridColumnSpan?: number
}
```

**テンプレートを動的レンダリングに変更する**:
- `editorType` を Phase 2 で作ったレジストリ（またはヘッダ用のラッパー）で解決する
- `MasterCombobox` など既存コンポーネントはそのまま使い、呼び出し方だけ変える

**動作確認**: 既存の5項目の Enter チェーンが壊れないこと。

### ステップ 3.2: Enter 移動順を統合する

- ヘッダの最後の項目からグリッドの先頭列への接続を Spec で表現する
  （例: `afterHeader: 'grid:productCode'`）
- グリッド内の Enter 移動は、Phase 1 の `editChainColIds` を `navigationSpec` に移動する
- コンポーザブルは Spec だけを見て動くようにする

### ステップ 3.3: F キーの割り当てを Spec 化する

- `F1`〜`F12` のマッピングをオブジェクトとして定義する
- 共通のデフォルト（新規・保存・読込など）を用意し、画面 Spec で上書きできるようにする
- ウィンドウの `keydown` リスナーはエンジンが1つだけ持ち、現在の画面の Spec を参照して処理を振り分ける

### ステップ 3.4: 別の業務画面を試しに作る（推奨）

**これが Phase 3 の本当のゴール確認**。

- 新しいルート（例: `/purchase/new`）を追加する
- **同じエンジンコンポーネント**に **別の Spec** を渡すだけで画面が立ち上がることを確認する
- 行データの型が異なる場合は、別の Row 型と別の commitSpec ルールを用意すればよい（過度な共通化は不要）
- 重要なのは **Vue ファイルを丸ごとコピーしなくてよい**こと

### ステップ 3.5: 疑似2段表示（任意 ― 仕様が固まっている場合のみ）

- `screen-spec-plan.md` セクション 4 に基づき、1行のまま上下段を表現するセルレンダラを field type や列オプションで指定できるようにする
- まだ仕様が固まっていなければ Phase 3 の完了条件からは除外し、別タスクにしてよい
