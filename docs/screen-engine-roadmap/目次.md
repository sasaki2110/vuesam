# 共通エンジン + Spec 実装ロードマップ（目次）

## このフォルダは何？

このフォルダには、**画面を量産できる仕組みを作るための作業計画**が入っています。

| ドキュメント | 内容 |
|---|---|
| [screen-spec-plan.md](../screen-spec-plan.md) | **何を作るのか・なぜ作るのか**（全体の設計方針・用語の定義） |
| 本フォルダ（`目次.md` + `phase*.md`） | **どうやって・どの順番で作るのか**（具体的な作業手順とチェックリスト） |

設計の考え方や用語は `screen-spec-plan.md` に合わせます。作業の段取りや完了基準はこちらが正です。

---

## 全体像：4つのフェーズ

今の `OrderNewView.vue` は、画面レイアウト・キー操作・データ処理がすべて1ファイルに詰め込まれています。このままだと、似た画面（発注・生産計画など）を追加するたびに巨大なファイルをコピペすることになり、保守が破綻します。

そこで「**どの画面でも共通の処理（エンジン）**」と「**画面ごとに変わる設定（Spec）**」に分離し、新しい画面は Spec ファイルを追加するだけで作れる状態を目指します。

この変更を4段階に分けて、少しずつ安全に進めます。

| 順番 | ファイル | やること | ゴール | 前提 |
|------|----------|----------|--------|------|
| 1 | [phase1.md](./phase1.md) | 既存コードの分解 | キー操作やフォーカス移動の処理をエンジンとして切り出す。**見た目や操作感は一切変えない** | なし |
| 2 | [phase2.md](./phase2.md) | 入力部品と副作用の整理 | セルエディタを「型名で呼び出す仕組み」にし、値確定時の処理（金額計算など）を宣言的に書けるようにする | Phase 1 完了 |
| 3 | [phase3.md](./phase3.md) | 画面定義の完全 Spec 化 | ヘッダ項目・Enter 順・F キーの割り当てまで、すべて設定データで表現できるようにする。**新画面 = 新 Spec ファイル**の状態にする | Phase 2 完了 |
| 4 | [phase4.md](./phase4.md) | 外部ファイル化（任意） | Spec を JSON/YAML に切り出し、コードを書かなくても画面定義を変更できるようにする | Phase 3 安定後 |

---

## 進め方のルール

1. 各 Phase の「完了条件」を満たすまで、書いてある順番で作業する
2. **Phase をまたいで一気に書き換えない**。特に Phase 1 は「動作が今と完全に同じ」が最優先。リネームや汎用化は Phase 2 以降で行う
3. 実装中に設計上の判断が必要になったら、**`decisions.md` に日付と決定内容を記録**する（1〜3行で十分）。コードとドキュメントの食い違いを防ぐため
