# Phase 4（任意）: Spec を外部ファイル化し、コードを書かずに画面定義を変更できるようにする

## このフェーズの目的

**一言で言うと**: Phase 3 までで TypeScript のオブジェクトとして安定した Spec を、JSON や YAML の外部ファイルに切り出す。これにより、エンジニア以外でも画面定義の変更ができるようになる。

**注意**: Phase 4 は必須ではありません。Phase 3 までの状態で十分な場合はスキップできます。このファイルは「将来やるならここまで」の参考として残します。

---

## 前提

- Phase 3 が完了していること（1つの画面が「エンジン + Spec」で構成され、2つ以上の業務画面で実証済み）

---

## 完了条件（Phase 4 を採用する場合のみ）

1. **単一の定義ファイルから画面が生成できる**
   - 例: `specs/order-new.yaml` を編集すると、ヘッダのラベル・列の表示名・Enter の移動順のうち少なくとも2つが反映される

2. **プログラム的な処理はハンドラ ID で接続されている**
   - 参照ダイアログ、複雑なロット引当などの処理は、Spec 側では ID で参照するだけにし、実装は別ファイルに置く
   - 外部ファイルに「コードそのもの」は書かない

3. **エラー時のフォールバックが文書化されている**
   - 外部 Spec の読み込みや生成に失敗したときの挙動（ビルドエラーで止める / デフォルト Spec を使う）が明記されている

---

## 作業手順

### ステップ 4.1: 外部ファイル化する範囲を決める

以下のうち、どこまでを外部ファイルに出すかを選ぶ。採用しなかったものは理由を `decisions.md` に残す。

| 対象 | 外部化するか |
|------|-------------|
| ヘッダ項目（ラベル・入力部品の種類） | |
| グリッド列（field, headerName, editable, fieldType） | |
| Enter の移動順（navigationSpec） | |
| F キーの割り当て（keySpec） | |

**外部化しないもの**: commitSpec のルール本体（JavaScript の処理）や複雑な valueGetter。これらはハンドラ ID だけを外部ファイルに書き、実装は TypeScript に残す。

### ステップ 4.2: バリデーションの仕組みを作る

- JSON Schema や Zod などで外部 Spec ファイルの構造を検証する
- エンジンが起動する前に不正な定義を検出してエラーにする（実行時にクラッシュするより安全）

### ステップ 4.3: ビルドパイプラインに組み込む

方式は2つのどちらかを選ぶ:
- **方式 A**: `pnpm specs:compile` コマンドで JSON/YAML → TypeScript モジュールを生成し、`src/generated/specs.ts` から import する
- **方式 B**: Vite プラグインを使い、JSON ファイルを直接 import できるようにする

### ステップ 4.4: 運用ルールを決める

- Spec 変更時のレビュー観点（キー名の破壊的変更、ハンドラの削除など）を `decisions.md` や CONTRIBUTING に短く記載する

---

## スコープ外（やらないこと）

- **すべての業務ロジックを独自言語（DSL）化する** → コストが高い割にメリットが少ない
- **信頼できないネットワークから Spec をそのまま実行する** → セキュリティ上の問題がある

---

## メンテナンスについて

Phase 4 を採用しなかった場合でも、このファイルは削除せず「将来の選択肢」として残しておきます。
