# 画面開発 手引書（人間向け）

本ドキュメントは、vuesam プロジェクトで **新しい業務画面を追加する** 際に、
人間が担当する作業の手引書です。

コードの実装は AI に依頼します。人間の仕事は **画面の要件を Markdown で定義し、
バックエンドの API 契約を整え、AI に実装を依頼し、結果を確認する** ことです。

> **関連ドキュメント**
>
> - [guide-for-ai.md](./guide-for-ai.md) — AI 向けの実装リファレンス（人間が読む必要はありません）

> - [archive/screen-engine-roadmap/phase4.md](./archive/screen-engine-roadmap/phase4.md) — Phase 4 の詳細（語彙表の出典）

## 1. 全体ワークフロー

画面を 1 つ追加する流れは、**人間系の作業** と **AI系の作業** に分かれます。

```
┌──────────────────────────────────────────────────────────────────────┐
│  人間系の作業                                                        │
│                                                                      │
│  ① 業務要件を整理する                                               │
│     └─ どんな画面が必要か？ 項目は？ 操作は？                       │
│                                                                      │
│  ② 画面仕様書を Markdown で書く                                     │
│     └─ docs/screen-specs/<画面ID>.md を作成                          │
│     └─ テンプレートをコピーして項目を埋める                          │
│                                                                      │
│  ③ バックエンド API を設計・共有する                                 │
│     └─ マスタ API、登録 API、一覧 API、削除 API                     │
│     └─ 400 エラーの JSON 契約を合意する                              │
│                                                                      │
├───────────────── Markdown を AI に渡す ──────────────────────────────┤
│                                                                      │
│  AI系の作業（AI が実行）                                             │
│                                                                      │
│  ④ 画面定義 Markdown を読む                                         │
│  ⑤ TypeScript Spec / ColDef ビルダ / CommitRule を生成              │
│  ⑥ ページ Vue を生成                                                │
│  ⑦ API クライアント関数を追加                                       │
│  ⑧ ルートを追加し、npm run build で確認                             │
│                                                                      │
├───────────────── 結果を人間が確認 ───────────────────────────────────┤
│                                                                      │
│  人間系の作業（確認）                                                │
│                                                                      │
│  ⑨ 画面を操作して動作を確認する                                     │
│  ⑩ 必要があれば追加依頼・修正依頼を AI に出す                       │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

**ポイント**: 人間はコードを書く必要はありません。Markdown の画面定義が「設計書」であり、
AI への「発注書」です。

## 2. 画面の種類を理解する

本基盤が対応する画面は **2 種類** です。

### 2.1 登録画面

ヘッダ入力 + 明細グリッド + 保存の画面。業務伝票の入力に使います。

```
┌────────────────────────────────────────────┐
│  画面タイトル                     [F1][F12] │
├────────────────────────────────────────────┤
│  契約先: [▼ 1001 山田商事]                 │
│  納入先: [▼ 3001 田中物流]                 │
│  納期:   [2026-04-07]                      │
├────────────────────────────────────────────┤
│  │ 行 │ 製品コード  │ 製品名 │ 数量│ 単価│ │
│  │  1 │ A001 ネジM3 │ ネジM3 │  100│   50│ │
│  │  2 │ B002 板金   │ 板金   │   20│  300│ │
│  │  3 │             │        │     │     │ │
│  │ ...│             │        │     │     │ │
└────────────────────────────────────────────┘
```

**特徴**:
- ヘッダ → グリッドを Enter で移動できる
- グリッド内でも Enter で右の列に移動、最終列で次行へ
- セル確定時に自動計算や名称の自動入力が走る
- F12 で保存（API に POST）

**既存の実装例**: [order-new.md](./screen-specs/order-new.md)（受注登録）、[purchase-new.md](./screen-specs/purchase-new.md)（発注登録）

### 2.2 一覧画面

検索条件 + 結果一覧の画面。データの検索・閲覧・削除・詳細遷移に使います。

```
┌────────────────────────────────────────────┐
│  画面タイトル                     [F1][F12] │
├────────────────────────────────────────────┤
│  受注番号: [        ] 契約先: [▼        ]  │
│  納期From: [        ] 納期To: [          ] │
├────────────────────────────────────────────┤
│  □│ 受注番号 │ 行 │ 製品   │ 数量│ 合計  │ │
│  □│ ORD-001  │  1 │ ネジM3 │ 100 │10,000 │ │
│   │          │  2 │ 板金   │  20 │ 6,000 │ │
│  □│ ORD-002  │  1 │ ボルト │  50 │ 5,000 │ │
│  │ ...                                     │
└────────────────────────────────────────────┘
```

**特徴**:
- グリッドは読み取り専用（編集しない）
- 行をダブルクリックで登録画面に遷移
- 行を選択して削除
- 同一伝票の行は Row Spanning で縦結合表示できる
- F12 で検索、F1 で検索条件クリア

**既存の実装例**: [order-list.md](./screen-specs/order-list.md)（受注一覧）

## 3. 画面仕様書（Markdown）の書き方

### 3.1 作業の流れ

1. `docs/screen-specs/` に `<画面ID>.md` ファイルを作成する
2. 既存のファイルをコピーして項目を書き換えるのが最も手軽
3. 下記のテンプレートに従ってセクションを埋める

### 3.2 登録画面テンプレート

```markdown
# 画面: ＜画面名＞

- **画面 ID**: `<screen-id>`
- **ルート**: `/<resource>/new`
- **初期明細行数**: ＜数値＞

## ヘッダ

| 項目 | 入力方式 | バリデーション | 備考 |
|------|---------|--------------|------|
| ＜項目名＞ | ＜入力方式＞ | ＜ルール or —＞ | ＜補足＞ |

## 明細グリッド

| 列名 | 型 | 編集 | バリデーション | 備考 |
|------|-----|------|--------------|------|
| 行 | 行番号 | × | — | 左固定 |
| ＜列名＞ | ＜型＞ | ○ or × | ＜ルール or —＞ | ＜補足＞ |

## 値確定ルール（副作用）

- ＜トリガー列名＞確定 → ＜処理内容＞

## F キー

- F1: 新規（共通デフォルト）
- F12: 保存（共通デフォルト）
- F＜N＞: ＜画面固有の機能＞

## マスタ・データソース

| 用途 | API | レスポンス形式 | 備考 |
|------|-----|--------------|------|
| ＜何の候補リスト＞ | GET /api/masters/＜name＞ | `[{ code, name }]` | |

## 画面アクション → API

| アクション | API | ボディ概要 | レスポンス概要 |
|-----------|-----|-----------|--------------|
| 保存（F12） | POST /api/＜resource＞ | `{ ..., lines: [...] }` | `{ id, message }` |

## 特記事項

- ＜例外的なルールや補足事項＞
```

### 3.3 一覧画面テンプレート

```markdown
# 画面: ＜画面名＞一覧

- **画面 ID**: `<screen-id>-list`
- **ルート**: `/<resource>`

## 検索条件

| 項目 | 入力方式 | 備考 |
|------|---------|------|
| ＜項目名＞ | ＜入力方式＞ | ＜補足＞ |

## 結果一覧

| 列名 | 表示形式 | 配置 | 備考 |
|------|---------|------|------|
| ＜列名＞ | ＜テキスト/数値/日付＞ | 左 or 右 | ＜補足＞ |

## 行操作

- ダブルクリック → ＜遷移先画面＞
- 削除 → DELETE /api/＜resource＞/{id}

## 検索 API

| アクション | API | パラメータ概要 | レスポンス概要 |
|-----------|-----|--------------|--------------|
| 検索 | GET /api/＜resource＞ | ＜クエリパラメータ＞ | JSON 配列 |

## F キー

- F1: 検索条件クリア
- F12: 検索実行

## 特記事項

- ＜補足＞
```

### 3.4 書き方のコツ

#### ヘッダの記入ルール

- **表の行の順番** がそのまま **Enter の移動順** になります
- Enter 順から外したい項目は備考に「Enter 順から除外」と書いてください
- 入力方式は自然な日本語で大丈夫です（揺れは AI が吸収します）

#### 入力方式の語彙

正確な用語を覚える必要はありません。以下のような自然な表現で書けば AI が解釈します。

| あなたが書く表現 | AI が解釈する型 |
|---|---|
| マスタ選択、コンボ、プルダウン、取引先選択 | ヘッダ: `masterCombobox` / グリッド: `codeAutocomplete` |
| テキスト、文字入力 | `text` |
| 数値、数量、金額 | `numeric` |
| 日付 | `date` |
| コード選択（候補リスト付き） | `codeAutocomplete` |
| 表示のみ、自動計算、読み取り専用 | `readOnlyText` or `readOnlyComputed` |

#### バリデーションの書き方

バリデーション列には以下のように書きます。

| あなたが書く表現 | 意味 |
|---|---|
| 必須 | 空欄不可 |
| — | チェックなし |
| 必須（入力行のみ） | 空行はスキップし、何か入力がある行だけチェック |
| N 文字以上 | 最小文字数チェック |
| N 文字以下 | 最大文字数チェック |
| 1 以上 | 数値の最小値チェック |
| 正規表現: ＜パターン＞ | パターンマッチ |
| ＜自由記述＞ | AI がカスタムバリデーションとして実装 |

#### 一覧の行スパン（Row Spanning）

一覧画面で、1 件の伝票が複数明細行になる場合（受注1件が明細3行など）、
ヘッダ系の列を縦結合して見やすくできます。

**書き方**: 結果一覧テーブルの備考に「行スパン」または「縦結合」と書くだけです。

```markdown
| 列名 | 表示形式 | 配置 | 備考 |
|------|---------|------|------|
| 受注番号 | テキスト | 左 | 行スパン |
| 行 | 数値 | 左 | |
| 製品名 | テキスト | 左 | |
| 合計 | 数値 | 右 | 行スパン |
```

ヘッダ系の列（受注番号、契約先、合計など同一伝票で値が同じ列）に「行スパン」を付けると、
同じ伝票内で縦結合されます。

#### API が未定の場合

画面アクション → API やマスタの API がまだ決まっていない場合は「（未定・モック）」と書いてください。
AI はモックデータで仮実装します。API が確定した後に差し替えを依頼できます。

```markdown
| 保存（F12） | （未定・モック） | — | — |
```

## 4. バックエンドに伝えるべき API 契約

バックエンド担当者と合意すべき **3 つの契約** があります。

### 4.1 マスタ API の形式

プルダウンやコード選択の候補リストに使うマスタ API は、以下の形式を守ってください。

```
GET /api/masters/<name>
Response: [{ "code": "string", "name": "string" }]
```

これがフロントの `CodeMasterItem` 型に対応します。

### 4.2 業務データ API のパターン

| 操作 | 方式 | パス例 | リクエスト | レスポンス |
|---|---|---|---|---|
| 登録 | POST | `/api/orders` | ヘッダ項目 + `lines: [明細]` | `{ id, message }` |
| 一覧検索 | GET | `/api/orders?key=value` | クエリパラメータ | JSON 配列 |
| 削除 | DELETE | `/api/orders/{id}` | なし | 204 No Content |

**一覧 API の重要なルール**: レスポンスは **1要素＝明細1行** のフラット配列にしてください。
ヘッダ情報（伝票番号、取引先名、合計金額など）は全明細行で繰り返し含めます。
これによりフロントで Row Spanning（行スパン）が正しく動作します。

### 4.3 400 エラーレスポンスの契約（最重要）

フロントのバリデーション表示と連携するため、400 エラーは以下の JSON 形式で返してください。

```json
{
  "code": "VALIDATION_ERROR",
  "message": "入力内容に誤りがあります",
  "details": [
    { "field": "contractPartyCode", "reason": "契約先コードは必須です" },
    { "field": "lines[0].productCode", "reason": "製品コードは必須です" }
  ]
}
```

**`field` の命名規約**:

| 対象 | パス形式 | 例 |
|---|---|---|
| ヘッダ項目 | リクエストボディのプロパティ名 | `contractPartyCode` |
| 明細項目 | `lines[インデックス].フィールド名` | `lines[0].productCode` |

**2 種類のエラー**:

| ケース | `code` | `details` | フロントの表示 |
|---|---|---|---|
| 入力バリデーション | `VALIDATION_ERROR` | エラー配列 | 各項目に赤枠 + メッセージ |
| 業務ルールエラー | `BAD_REQUEST` | 空配列 `[]` | `message` をアラート表示 |

**`reason` は日本語で**: フロントは `reason` の文字列をそのまま表示します。
Spring Boot の場合、Bean Validation の `message` を日本語に設定してください。

### 4.4 バックエンド担当者に渡すチェックリスト

新しい画面のバックエンド API を依頼する際は、以下をセットで渡してください。

```
□ マスタ API
  - パス: GET /api/masters/<name>
  - レスポンス: [{ code, name }]

□ 登録 API
  - パス: POST /api/<resource>
  - リクエスト: { ヘッダ項目, lines: [明細項目] }
  - レスポンス: { id, message }
  - 400 エラー: 上記の JSON 契約に準拠

□ 一覧 API
  - パス: GET /api/<resource>
  - レスポンス: 明細フラット配列（1要素＝明細1行、ヘッダ情報は繰り返し）
  - 名称列は JOIN で解決して含める（コードだけでなく名前も）

□ 削除 API
  - パス: DELETE /api/<resource>/{id}
  - レスポンス: 204
  - ヘッダ＋明細をトランザクションで削除

□ 共通
  - Content-Type: application/json
  - 認証: Authorization: Bearer {token}
  - 日付: yyyy-MM-dd（ISO 8601）
  - CORS: フロント開発サーバのオリジンを許可
```

## 5. AI への依頼の仕方

### 5.1 基本の依頼文

画面仕様書を書いたら、AI に以下のように依頼します。

> `docs/screen-specs/<画面ID>.md` を読んで、画面を追加してください。

これだけで AI は以下を生成します:

| 生成対象 | 説明 |
|---|---|
| `src/features/<screen>-screen/<screen>NewSpec.ts` | 画面 Spec（ヘッダ、ナビ、バリデーション） |
| `src/features/<screen>-screen/<screen>Grid.ts` | グリッド列定義ビルダ |
| `src/features/<screen>-screen/<screen>CommitRules.ts` | セル確定時の副作用ルール |
| `src/features/<screen>-screen/<screen>Types.ts` | 行データの型 |
| `src/views/<Screen>NewPage.vue` | ページ Vue |
| `src/api/client.ts` への追加 | API 呼び出し関数 |
| `src/router/index.ts` への追加 | ルート定義 |

一覧画面の場合:

| 生成対象 | 説明 |
|---|---|
| `src/features/<screen>-screen/<screen>ListSpec.ts` | 一覧 Spec |
| `src/features/screen-engine/listScreenSpecRegistry.ts` への追加 | Spec 登録 |
| `src/api/client.ts` への追加 | 検索・削除の API 関数 |
| `src/router/index.ts` への追加 | ルート定義 |

### 5.2 AI が参照するドキュメント

AI は依頼を受けると、自動的に以下を参照して実装します:

- あなたが書いた画面定義 Markdown（`docs/screen-specs/<画面ID>.md`）
- AI 向けリファレンス（`docs/guide-for-ai.md`）
- 既存の実装例（`src/features/order-screen/` や `src/features/purchase-screen/`）

**`guide-for-ai.md` を AI に読ませる**: 初回の依頼時や、AI が不慣れに見える場合は、
依頼文に「`docs/guide-for-ai.md` も参照してください」と付け加えてください。

### 5.3 よくある追加依頼のパターン

| やりたいこと | 依頼例 |
|---|---|
| 列を追加 | 「出荷画面の明細に『ロット番号』列を追加してください。テキスト入力、必須です」 |
| 確定ルールの変更 | 「製品コード確定時に、在庫数も自動表示してください」 |
| 一覧画面の追加 | 「出荷一覧画面を追加してください。仕様は `docs/screen-specs/shipping-list.md` にあります」 |
| バリデーションの追加 | 「数量に上限 9999 のチェックを追加してください」 |
| API の差し替え | 「保存のモックを本物の API（POST /api/shipments）に差し替えてください」 |
| Enter 順の変更 | 「備考を Enter チェーンから外してください」 |

### 5.4 登録画面と一覧画面をセットで追加する場合

1. まず `docs/screen-specs/<画面ID>.md`（登録）を書く
2. 次に `docs/screen-specs/<画面ID>-list.md`（一覧）を書く
3. AI に「両方読んで画面を追加してください」と依頼

## 6. 実装後の確認ポイント

AI が実装を完了したら、以下を確認してください。

### 6.1 ビルド確認

```bash
npm run build
```

型エラーがなければ OK です。エラーがあれば AI に修正を依頼してください。

### 6.2 画面動作の確認チェックリスト

**登録画面**:

- [ ] 画面が表示される
- [ ] ヘッダ項目が仕様書どおりに並んでいる
- [ ] ヘッダで Enter を押すと次の項目に移動する
- [ ] ヘッダ最終項目で Enter → グリッドの最初の編集列にフォーカス
- [ ] グリッドで Enter を押すと右の列に移動する
- [ ] 最終列で Enter → 次行の最初の編集列に移動（行追加される）
- [ ] 値確定ルールが動く（自動計算、名称の自動入力など）
- [ ] F12 で保存できる（API が実装済みの場合）
- [ ] 必須項目を空にして保存 → バリデーションエラーが赤枠で表示される
- [ ] F1 で画面がクリアされる

**一覧画面**:

- [ ] 画面が表示される
- [ ] F12 で検索結果が表示される
- [ ] 行スパンが正しく結合されている（設定した場合）
- [ ] 数値列が右寄せで表示されている（設定した場合）
- [ ] 行をダブルクリック → 登録画面に遷移する
- [ ] 行を選択して削除 → 確認ダイアログ → 削除される（API が実装済みの場合）
- [ ] F1 で検索条件がクリアされる

### 6.3 問題があった場合

AI に以下のように報告してください:

> 「○○画面で、Enter を押しても数量列に移動しません。修正してください」
> 「一覧画面で、金額が左寄せになっています。右寄せにしてください」
> 「バリデーションエラーが出ません。製品コードを空にして保存しても赤枠が出ません」

具体的な症状を伝えると、AI が素早く修正できます。
