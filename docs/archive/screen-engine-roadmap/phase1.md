# Phase 1: 既存画面の分解 ― エンジンと Spec に分ける（動作は変えない）

## このフェーズの目的

**一言で言うと**: `OrderNewView.vue` に全部入りになっている処理を、「共通で使い回せる部分（エンジン）」と「受注画面固有の設定（Spec）」に分離する。

具体的には:

- **エンジンに移すもの**: キー操作の制御、Enter でのフォーカス移動、AG Grid との連携処理など、どの業務画面でも同じ仕組みで動く部分
- **Spec に残すもの**: 列の並び順、画面タイトル、マスタデータの種類など、画面ごとに内容が変わる設定データ
- **絶対条件**: ユーザーから見た操作感（F01/F12、ヘッダの Enter 移動、明細の Enter 移動、オートコンプリートとの競合回避、保存処理）は **Phase 1 の前後で完全に同じ**であること

---

## 完了条件（これが全部満たされたら Phase 1 は終わり）

1. **`OrderNewView.vue` が薄くなっている**
   - 主な責務は「どの Spec を使うか」「マスタデータを渡す」だけ
   - キー制御やフォーカス移動のロジックの大半はエンジン側のファイルに移っている
   - テンプレートとスタイルはそのままで構わない

2. **以下の処理が独立したモジュールにまとまっている**（テストや検証がしやすい状態）
   - AG Grid で編集中に Enter キーを横取りしない制御（`suppressKeyboardEvent` 相当）
   - 編集終了後に次のセル・次の行・新しい行へ移動する処理
   - F01（新規）/ F12（保存）のキーボードショートカット

3. **Enter 移動に使う列の情報が Spec から取得できる**
   - 現状の `ORDER_EDIT_COLS` / `ORDER_ENTER_NAV_COL_KEYS` のようなハードコード定数ではなく、Spec（または Spec から生成されるデータ）として定義されている

4. **ビルドが通り、既存の画面が壊れていない**
   - `npm run build` が成功する
   - `/orders/new` と `/orders/new-alt` の両方が従来どおり動く

---

## やらないこと（Phase 1 の範囲外）

- 型名 `OrderLineRow` やフォルダ名 `order-screen` のリネーム → Phase 2〜3 で必要になったときにまとめてやる
- 疑似2段セルや `個別納期` / `備考` 列の追加 → 設計プラン上の将来例であり、今のコードにないものは追加しない
- Phase 1 はあくまで **今あるコードの切り出し・整理**が目的

---

## 作業手順（この順番で進める）

### ステップ 1.1: 現状コードの仕分け

`OrderNewView.vue` の `<script setup>` にある関数やロジックを、以下の3つに分類する。

| 分類 | 内容 | 例 |
|------|------|-----|
| **A. エンジン候補** | どの業務画面でも使い回したい処理 | Enter キー抑制、編集後フォーカス移動、gridApi 管理、F01/F12 振り分け |
| **B. 受注デモ固有** | この画面だけに必要な処理 | 保存時のデータ形式、`PARTIES`/`PRODUCTS` の直接参照、ヘッダの ref 名 |
| **C. 汎用ユーティリティ** | どこにでも置ける小さな処理 | 日付フォーマットなど（エンジンか `utils` かは好みで決める） |

### ステップ 1.2: Enter 移動ロジックの切り出し（第一弾）

**新しいファイルを作る**: 例 `src/features/order-screen/useOrderGridEnterNav.ts`
（名前は自由だが「グリッドの Enter 移動」だとわかる名前にする）

**移す処理**:
- `advanceRowOnStop` ref と、それに依存する Enter キー抑制・次セル移動のロジック
- 引数として「Enter で順番に移動する列の一覧」と「Enter で編集を終了する列の一覧」を受け取る形にする（今ある定数をそのまま渡せばよい）

**Vue 側の変更**:
- コンポーザブルを呼び出し、返り値を `defaultColDef` と `@cell-editing-stopped` に接続する

**動作確認のポイント**:
- 製品コードのリスト表示中に Enter が奪われないか
- 数量→単価→次行の製品コードと Enter で順に移動するか
- 最終行の単価で Enter すると新しい行ができ、その行の製品コードに入るか

### ステップ 1.3: 列の並び順を Spec に持たせる

`orderNewSpec.ts` に以下の情報を追加する。

| 追加する項目 | 元の定数 | 役割 |
|---|---|---|
| `editChainColIds: string[]` | `ORDER_EDIT_COLS` | Enter で順番に移動する列 ID の配列 |
| `enterStopEditingColIds: string[]` | `ORDER_ENTER_NAV_COL_KEYS` | Enter で編集終了とみなす列 ID の配列 |

- `order-new` と `order-new-alt` で値が異なる場合は alt 用に上書きする。同じなら共通で1か所に定義すればよい
- Vue 側はこの値をコンポーザブルに渡すだけにする

### ステップ 1.4: F キー処理の切り出し（推奨・任意）

ウィンドウの `keydown` で F01/F12 を拾っている処理を、`useScreenFunctionKeys.ts` のような独立ファイルに分離する。

- `{ onNew, onSave }` のようにハンドラを外から注入できる形にする
- こうしておくと Phase 3 で `keySpec`（F キーの割り当て設定）に接続するときスムーズ

### ステップ 1.5: 全体の動作確認

下の「手動チェックリスト」をすべて確認し、問題があれば Phase 1 の中で修正する。

---

## 手動チェックリスト

- [ ] 初回表示で「契約先」にフォーカスがある
- [ ] ヘッダで Enter を押すと、契約先 → 納入先 → 納入場所 → 納期 → 内示 → 明細先頭（製品コード編集開始）の順に移動する
- [ ] 納期が未入力の状態でフォーカスすると +7 日の値が自動で入る
- [ ] 明細の製品コードでサジェストリストが開いている間、Enter がグリッドに奪われない
- [ ] 数量で Enter → 単価へ移動。単価で Enter → 次の行の製品コードへ移動する
- [ ] 最後の行の単価で Enter すると、新しい行が追加され、その行の製品コードに入る
- [ ] 製品コードを確定すると品名が自動で埋まり、数量セルにフォーカスが移る
- [ ] 数量・単価を変更すると金額が再計算される
- [ ] F01 でヘッダ・明細が初期化され、契約先にフォーカスが戻る
- [ ] F12 でコンソールログ・アラート・初期化が行われる（現状どおり）
- [ ] `order-new-alt` で末尾の読み取り専用列が表示され、既存の動作を壊していない

---

## 注意すべきリスクと対策

| リスク | 何が起きるか | 対策 |
|--------|------------|------|
| gridApi のタイミングずれ | コンポーザブル化で `gridApi` の取得タイミングが変わり、行追加後のフォーカスが失敗する | ステップを細かくコミットし、ステップ 1.2 の直後に「最下行→行追加」の動作確認を最優先で行う |
| 列 ID の不一致 | `suppressKeyboardEvent` で使う列 ID がずれて Enter 制御が壊れる | `colId` と `field` を混同しない。現状と同じ `getColId()` ベースを維持する |

---

## Phase 2 に渡すもの

Phase 1 が終わった時点で、エンジン側に「どの列を Enter で巡回するか」の情報が **Spec 経由で渡っている**ことが重要です。Phase 2 では、この配列を `navigationSpec` の基礎データとして読み替えて進化させます。
