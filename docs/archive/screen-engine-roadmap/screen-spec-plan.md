## 画面量産向け実装プラン（共通エンジン + 仕様 Spec）

### 目的
オフコン向けマイグレデモのような「キーボード中心（Enter / Fキー）」「入力確定時に副作用（計算・マスタ確定・参照）」を含む受注系画面を、**横展開（画面追加）しても保守コストが爆発しない**構造に寄せる。

現状の `OrderNewView` の実装は、以下が一つのファイルに密結合しています。
- レイアウト（ヘッダ部 / 明細グリッド部）
- キー仕様（F01/F12/Enterでのフォーカス移動）
- エディタ（ag-grid セルエディタ、Vue コンポーネントのオートコンプリート）
- 値確定時の副作用（productCode→productName / 数量・単価→金額）

横展開時に「仕様が変わる場所」と「変えない場所」を分離し、
**共通実装（共通エンジン）**と**仕様定義（Spec）**に切り出すのがゴールです。

---

## 0. 用語（このドキュメント内）
- **共通エンジン**: 画面の骨格・キー処理・コミット処理を提供する、再利用可能な実装。
- **Spec（仕様データ）**: 画面ごとに変わる項目・列・Enter移動・Fキー挙動・副作用などを宣言として持つデータ。
- **エディタ（Editor）**: 入力UI（例: productCode用 Autocomplete、数値入力、コード参照ダイアログなど）。
- **コミット（Commit）**: ユーザー操作で確定した値をグリッド/状態へ反映し、副作用を実行すること。

---

## 1. 現状から見える「変えるべき境界」

横展開で確実に変わるもの（＝Spec側に寄せる）：
1. ヘッダ項目（契約先コード/納入先コード/納入場所/納期/内示番号/…）
2. 明細グリッドの列構成（列追加/削除、1明細あたりの表示項目の増減）
3. 入力フロー（左→右、上→下が基本。ただし項目飛びや最終項目で次明細へ移動）
4. Fキーの種類と挙動（F01/F02/…/F12 は共通化しやすいが、画面固有F07/F08等が追加される）
5. エディタ確定時の副作用（数量・単価→金額計算、コード確定で別フィールド更新、ロット引当ダイアログ等）

変えないもの（＝共通エンジン側に寄せる）：
1. 画面の骨格（ヘッダ領域、明細グリッド領域、ボタン配置、背景などのポリシー）
2. Enter/Fキーの“処理基盤”（キーイベント受け、編集状態を考慮し、フォーカス移動/コミットを制御）
3. コミットパイプライン（編集停止/値反映/副作用実行の一連の流れ）
4. エディタの登録/呼び出し（field type → Editor component の解決）

## 2. 「共通エンジン + Spec」の目標アーキテクチャ

### 2.1 画面生成のトップレベル
- 共通コンポーネント（例: `OrderScreenTemplate`）が以下を受け取る
  - `spec: OrderScreenSpec`
  - `dataSources: マスタ取得関数、参照ダイアログの呼び出し関数`
  - `actions: Fキーや画面固有処理のハンドラ（必要に応じて）`
- 画面ごとに `spec` を差し替えることで、UIと挙動が変わる。

### 2.2 Spec の中心モデル（推奨）
Specは「宣言 → 共通エンジンが解釈 → 実装を組み立てる」形にする。

推奨する Spec の最小構成：
1. `headerSpec`
   - 入力項目の配列（項目ID、ラベル、editor type、Enter遷移）
2. `gridSpec`
   - 1明細（logicalRow）の列定義
   - “疑似2段表示”の割当（見た目上の上下でどのフィールドを配置するか）
   - editable/readOnly の区分
3. `navigationSpec`
   - Enter の移動順（editableフィールド順）
   - 最終項目での次明細移動ルール
   - 項目飛び（例: 表示だけの項目は Enter移動対象外）
4. `keySpec`
   - F01〜F12 の共通コマンド定義
   - 画面固有F07/F08等の上書き/追加
5. `commitSpec`
   - フィールド確定時の副作用（宣言型）
   - 参照ダイアログ等の imperative handler（必要に応じて）

## 3. エディタ増殖問題の回避（Editor を増やし過ぎない）

画面数が増えるほど「項目が増える」＝「エディタ実装も増える」懸念が出ます。
これへの対策は、エディタを“項目”ではなく“field type”で設計することです。

例（field type の考え方）：
- `codeAutocomplete`（コード+名称のプルダウン/Autocomplete）
- `dialogLookup`（ダイアログで一覧表示して確定）
- `numeric`（数値入力）
- `date`（日付入力）
- `readOnlyComputed`（計算値の表示だけ、Enter移動対象外）

このとき画面固有のマスタ（倉庫マスタ、担当者マスタ等）が増えても、
Editor実装は増やさずに “取得元・ラベル・確定ルール” を props / handler として差し込むだけに寄せるのがポイントです。

## 4. 1明細2段（疑似2段表示）の仕様化（方式）

今回の確認結果より、本仕様の「疑似2段」は **AG Grid の row を増やさない**（＝物理2行にしない）方向に寄せます。

### 4.1 表示の考え方
- AG Grid 上は `1明細 = 1 row`
- row 内で “上段に見せたい項目” と “下段に見せたい項目” をセル描画/エディットUI側で縦方向に分けて表示する
- Enter移動は “見た目上の段” ではなく “論理フィールド順（editableフィールド順）” で制御する

### 4.2 Enter移動の基本（今回の画面仕様例）
- editable順は以下（表示だけの項目は除外）
  - `製品コード` → `数量` → `個別納期` → `備考` → 次明細の `製品コード`

さらにあなたの追加条件として、
- **1明細の最終項目（=2段目の最後＝備考）で Enter**した場合は
  - 次明細へ移動（次 row の先頭項目へフォーカス）

このルールを `navigationSpec` として宣言できれば、画面追加時に壊れにくくなります。

## 5. 共通エンジンの責務（キー/コミット）

### 5.1 Enter制御（編集状態と競合回避）
AG Grid と Vue コンポーネントのオートコンプリートのような“ポップアップ系”は、Enterが競合しやすいです。
共通エンジンは少なくとも次を保証する必要があります。
- 編集中の Enter は、ポップアップ候補確定と衝突しないように制御する
- ポップアップが開いている間は、Enterのグリッド処理を抑止し、エディタ側の確定に任せる
- 確定後に編集終了し、コミットパイプラインへ渡す

（現状コードでは `suppressKeyboardEvent` と “リストが開いているか” の判定がその役割を担っています。これを共通化して再利用します。）

### 5.2 Fキー基盤
- F01: 新規（初期化）
- F12: 確定/保存（コミットを経由）
- F02: 読込（必要なら外部データ投入）
- F04/F05/F06/F11/F07/F08…: 画面固有・共通に基づき handler を差し替え

実装としては
- `keySpec` から “Fキー→実行する命令（action）” を解決
- 共通エンジンが “現在の編集状態” を考慮してから handler を呼ぶ
が安定します。

### 5.3 コミットパイプライン（副作用）
- フィールド確定イベント（value changed / editing stopped等）を受ける
- `commitSpec` に従い副作用を適用
  - 例：`quantity/unitPrice` 確定で `amount = quantity * unitPrice`
  - 例：`productCode` 確定で `productName` 更新
  - 例：ロット確定で関連フィールドを更新（ダイアログ確定ハンドラ）
- 最後にグリッドへ反映し、Enterの次移動を許可する

ここを “一箇所” に集約しないと、画面追加時に副作用実装が散って追いづらくなります。

## 6. 実装フェーズ（提案）

### Phase 1: 現行画面を Spec 化し、同等動作を再現
- `OrderNewView` を分解し、Specを作る（まずは列とキー、Enter順だけでも良い）
- 共通エンジンは既存のロジックと同じ挙動を満たす
- “現行と一致” を最優先する

### Phase 2: Editor type と commitSpec を整理
- productCode 用 editor のような既存部品を field type として整理
- quantity/unitPrice→amount のような副作用を宣言型として共通解釈に寄せる

### Phase 3: 画面固有差分だけ Spec で上書きできる状態へ
- ヘッダ項目追加
- 列追加/削除
- Enterの項目飛びや最終項目での次row移動
- F07/F08の差し込み

### Phase 4（必要なら）定義体→生成を拡張
- “列/ヘッダ/Enter遷移” を中心に生成対象を増やす
- ダイアログ処理など imperative が必要な部分は handler 参照で妥協し、生成範囲を限定する

**各フェーズの作業順・完了条件・チェックリスト**は [screen-engine-roadmap/目次.md](./screen-engine-roadmap/目次.md) を参照。

## 7. 期待できる効果
- 画面追加のたびに `OrderNewView.vue` 相当の巨大ファイルを複製しなくて済む
- Enter/Fキーの細かな競合（AG Grid とオートコンプリートUI）が “共通エンジン側” に閉じる
- 仕様差分は Spec に寄せられ、レビュー時の差分が小さくなる
