| D | [D-validation.md](./D-validation.md) | バリデーション | ヘッダ・明細の入力チェックと API 400 エラーのフィールド表示を共通基盤化する | B〜C 完了後 |
| — | [backlog.md](../../backlog.md) | 技術課題バックログ | 将来対応すべき課題の一覧（トリガー条件付き） | 随時更新 |

## 推奨する実施順序

```
B + C（一覧画面 + View 分割、同時進行） → D（バリデーション） → A（要件確定後に再検討）
```

- **A** は先送り。具体的な画面要件（列数・横幅不足の程度）が確定した段階で方式を選定する
- **B と C** は相互に依存する（一覧画面追加で View が肥大化するので、先に分割する）
- **D** は B〜C で画面構造が安定した後に横断的に入れる

---

## 進め方のルール

1. 各ステップの「完了条件」を満たすまで、書いてある順番で作業する
2. 実装中に設計上の判断が必要になったら、[decisions.md](../screen-engine-roadmap/decisions.md) に日付と決定内容を記録する
3. `npm run build` が成功し、既存動作が壊れないことを各ステップで確認する
