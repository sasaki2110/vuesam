# フロント開発向け：最小接続実装（Vue / Docker 開発環境想定）

バックエンド（本リポジトリの Spring Boot API）と、Docker コンテナ内の Vue 開発サーバーとを、**最小限のコード**でつなぐための手順です。  
受注画面の本実装より前の「疎通・認証・1本の API 呼び出し」までを対象とします。

**注意:** ローカルにクローンした参照用リポジトリ（エディタ用）と、コンテナ内の実作業用ワークツリーは別物として扱います。このドキュメントは **コンテナ内で作業する開発者**が手元にコピーして使うことを想定しています。

---

## 1. 前提（バックエンド）

- API が起動していること（例: `http://localhost:8080`）
- 疎通確認: `GET /health` → `200` と JSON（認証不要）
- 業務 API は JWT（Bearer）前提
  - `POST /api/auth/login` … トークン取得（認証不要）
  - `GET /api/projects` など ` /api/**` … トークン必須

バックエンドの CORS は `http://localhost:5173` を許可する設定になっていることが多いです。  
ブラウザで開く **オリジン**（`http://localhost:5173` と `http://127.0.0.1:5173` は別）が一致しないと CORS エラーになります。開発時は **URL を統一**してください。

---

## 2. Docker 内フロントでよくある URL の考え方

ブラウザはホスト PC 上で動くため、`fetch` の向き先は **ブラウザから見て到達可能なホスト名**になります。

| 方式 | 概要 | メリット |
|------|------|----------|
| A. 直接バックエンドURL | `VITE_API_BASE_URL=http://localhost:8080` のように絶対 URL で呼ぶ | 設定が単純 |
| B. Vite プロキシ | フロントは相対パス `/api/...` のみ叩き、Vite がコンテナから `host.docker.internal:8080` 等へ転送 | ブラウザからはオリジンが 5173 のみになり、CORS を気にしにくい |

コンテナ内の Node がバックエンドに届ける必要がある場合、プロキシの **target** には `host.docker.internal`（Windows / Mac の Docker Desktop で一般的）や、ネットワーク上のホスト名を指定します。環境によって名前は異なるため、チームで決めてください。

---

## 3. 手順概要

1. 環境変数で API の基底 URL（またはプロキシ前提の相対パス）を決める  
2. `POST /api/auth/login` で JSON を送り、`accessToken` を取得する  
3. `sessionStorage` 等にトークンを保持（最小例。運用ルールに合わせて変更可）  
4. `Authorization: Bearer <token>` を付けて `GET /api/projects` を試す  

---

## 4. 環境変数（Vite）

プロジェクトルートに `.env.development`（またはチーム標準の env ファイル）を置きます。

**方式A（直接 URL）の例:**

```env
VITE_API_BASE_URL=http://localhost:8080
```

**方式B（プロキシ利用）の例:**

- `vite.config.ts` で `/api` をバックエンドにプロキシする  
- コード側は `VITE_API_BASE_URL` を空にするか、コード中で相対パス `''` として `fetch('/api/...')` だけ使う  

`import.meta.env.VITE_API_BASE_URL` から読み取ります。

---

## 5. 最小 API クライアント例（`fetch` のみ）

新規ファイル例: `src/api/client.ts`（パスはプロジェクトに合わせる）

```typescript
const baseUrl = (import.meta.env.VITE_API_BASE_URL as string | undefined)?.replace(/\/$/, '') ?? ''

function authHeader(): HeadersInit {
  const token = sessionStorage.getItem('accessToken')
  return token ? { Authorization: `Bearer ${token}` } : {}
}

export type LoginResponse = {
  accessToken: string
  tokenType: string
}

export async function login(username: string, password: string): Promise<LoginResponse> {
  const url = `${baseUrl}/api/auth/login`
  const res = await fetch(url, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username, password }),
  })
  if (!res.ok) {
    throw new Error(`login failed: ${res.status}`)
  }
  return (await res.json()) as LoginResponse
}

export async function fetchProjects(): Promise<unknown> {
  const url = `${baseUrl}/api/projects`
  const res = await fetch(url, { headers: authHeader() })
  if (!res.ok) {
    throw new Error(`projects failed: ${res.status}`)
  }
  return res.json()
}
```

**利用例（任意の画面や一時ボタンから）:**

```typescript
import { fetchProjects, login } from '@/api/client'

async function connectSample() {
  const { accessToken } = await login('demo', 'password')
  sessionStorage.setItem('accessToken', accessToken)
  const projects = await fetchProjects()
  console.info('projects', projects)
}
```

モック認証の想定ユーザー名・パスワードはバックエンドの `AuthController` に合わせます（例: `demo` / `password`）。

## 6. ログインリクエストの注意（JSON）

Windows / PowerShell で `curl.exe` を使う場合、インライン JSON が壊れやすいです。  
検証用は **ファイルに JSON を書いて `--data-binary "@request.json"`** を推奨します。

---

## 7. 方式B：Vite プロキシ設定例

`vite.config.ts` の `server.proxy` に概ね次のような設定を追加します（ターゲットホストは環境に合わせて変更）。

```typescript
export default defineConfig({
  // ...existing...
  server: {
    proxy: {
      '/api': {
        target: 'http://host.docker.internal:8080',
        changeOrigin: true,
      },
      '/health': {
        target: 'http://host.docker.internal:8080',
        changeOrigin: true,
      },
    },
  },
})
```

この場合、クライアントの `baseUrl` は空文字のまま `fetch('/api/auth/login')` のように **相対 URL** で呼びます。

## 8. トラブルシューティング

| 現象 | 確認すること |
|------|----------------|
| CORS エラー | ブラウザのアドレスが `allowedOrigins` と一致しているか（localhost vs 127.0.0.1） |
| ログインは通るが API が 401 | `Authorization` ヘッダの付与、`Bearer ` プレフィックス、トークンの格納キーの取り違え |
| 接続拒否 (ERR_CONNECTION_REFUSED) | バックエンド起動、ポート、Docker から見たホスト名（プロキシの `target`） |
| ログイン 401 | ユーザー名・パスワード、JSON 本文がサーバで解釈できているか |
